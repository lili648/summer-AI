# 内部工具用户认证系统 — 技术规格文档

**版本**: 1.0
**日期**: 2026-07-14
**状态**: 待实施

---

## 1. 概述与范围

### 1.1 系统名称

内部工具用户认证系统

### 1.2 技术栈

| 层级 | 选型 | 说明 |
|---|---|---|
| 前端框架 | Streamlit | 与现有应用一致 |
| 数据库 | MySQL | 已有环境 |
| 密码加密 | bcrypt | `pip install bcrypt`，自动加盐 |
| Python 驱动 | PyMySQL | 已有依赖 |

### 1.3 功能范围

**包含：**
- 用户注册（用户名 + 密码）
- 用户登录（验证凭证，建立会话）
- 用户登出（销毁会话）
- 登录态守卫（未登录用户自动跳转登录页）

**不包含（明确排除）：**
- 密码找回/重置
- 邮箱/手机验证
- 角色权限分级
- OAuth/第三方登录
- 验证码
- 操作审计日志

### 1.4 假设与约束

- 用户数为内部人员，不超过 100 人
- 单机部署，无横向扩展需求
- 服务重启后所有用户需重新登录（Streamlit Session 特性，可接受）
- 浏览器关闭即视为登出

## 2. 数据库设计

### 2.1 数据库信息

| 属性 | 值 |
|---|---|
| 数据库名 | `app_auth` |
| 字符集 | `utf8mb4` |
| 引擎 | InnoDB |

### 2.2 用户表 `users`

```sql
CREATE TABLE `users` (
    id          INT AUTO_INCREMENT PRIMARY KEY   COMMENT '主键ID',
    username    VARCHAR(50)  NOT NULL UNIQUE     COMMENT '用户名',
    password    VARCHAR(255) NOT NULL             COMMENT 'bcrypt密码哈希',
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
    last_login  DATETIME     DEFAULT NULL         COMMENT '最后登录时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 2.3 字段说明

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | INT | PK, AUTO_INCREMENT | 用户唯一标识 |
| `username` | VARCHAR(50) | NOT NULL, UNIQUE | 登录用户名，唯一 |
| `password` | VARCHAR(255) | NOT NULL | bcrypt 哈希结果（60字符），留足余量 |
| `created_at` | DATETIME | DEFAULT NOW | 账号创建时间 |
| `last_login` | DATETIME | NULLABLE | 最近一次登录时间，注册时为空 |

### 2.4 索引设计

| 索引名 | 字段 | 类型 | 用途 |
|---|---|---|---|
| `PRIMARY` | `id` | 主键 | 行标识 |
| `uniq_username` | `username` | UNIQUE | 登录时按用户名查找，同时保证唯一 |

## 3. 功能模块设计

### 3.1 模块划分

```
auth_app/
├── main.py              # 入口，登录态守卫 & 页面路由
├── auth.py              # 认证逻辑（注册、登录、登出）
├── database.py          # 数据库连接 & 用户 CRUD
└── pages/
    └── home.py          # 登录后的主功能页面（示例）
```

### 3.2 模块职责

#### `database.py` — 数据访问层

| 函数 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `get_conn()` | — | `Connection` | 获取 MySQL 连接 |
| `find_user(username)` | `str` | `dict \| None` | 按用户名查询，返回用户记录（含密码哈希） |
| `insert_user(username, password_hash)` | `str, str` | `None` | 插入新用户，重复用户名时抛异常 |
| `update_last_login(username)` | `str` | `None` | 更新最后登录时间 |

#### `auth.py` — 认证逻辑层

| 函数 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `hash_password(password)` | `str` | `str` | 调用 bcrypt 生成哈希 |
| `verify_password(password, hash)` | `str, str` | `bool` | 明文密码与哈希比对 |
| `register(username, password)` | `str, str` | `(bool, str)` | 注册，返回 (成功与否, 提示消息) |
| `login(username, password)` | `str, str` | `(bool, str)` | 登录，成功时写入 `st.session_state["user"]` |
| `logout()` | — | `None` | 清除 session，回到登录页 |
| `get_current_user()` | — | `str \| None` | 从 session 取当前用户名 |

#### `main.py` — 页面路由 & 守卫

- 调用 `get_current_user()` 判断登录状态
- **未登录** → 显示登录 / 注册两个 Tab
- **已登录** → 显示主功能页面，侧边栏展示用户名和登出按钮

### 3.3 注册流程

```
用户输入 username + password + confirm_password
  → 校验非空
  → 校验两次密码一致
  → 校验密码长度 ≥ 6 位
  → 校验用户名不重复（查库）
  → bcrypt 哈希密码
  → 写入 users 表
  → 提示注册成功，切换到登录 Tab
```

### 3.4 登录流程

```
用户输入 username + password
  → 校验非空
  → 按 username 查库
  → 未找到 → "用户名或密码错误"
  → 找到 → bcrypt 验证密码
  → 失败 → "用户名或密码错误"（不区分具体原因）
  → 成功 → st.session_state["user"] = username
          → 更新 last_login
          → 页面自动切换到主功能页
```

### 3.5 登出流程

```
用户点击登出按钮
  → 清除 st.session_state["user"]
  → st.rerun() 刷新页面回到登录页
```

## 4. 页面 UI 设计

### 4.1 登录页面

- 标题：🔐 内部工具用户认证
- 两个 Tab：登录（默认激活）/ 注册
- 登录 Tab：用户名输入框 + 密码输入框（type="password"）+ 登录按钮（type="primary"）
- 注册 Tab：用户名输入框 + 密码输入框 + 确认密码输入框 + 注册按钮（type="primary"）

### 4.2 主功能页面（登录后）

- 侧边栏顶部显示当前登录用户名
- 登出按钮，点击清除 session 回到登录页
- 主区域承载业务功能

### 4.3 错误提示规范

| 场景 | 提示方式 | 文案 |
|---|---|---|
| 用户名/密码为空 | `st.error` | "用户名和密码不能为空" |
| 登录验证失败 | `st.error` | "用户名或密码错误" |
| 注册密码不一致 | `st.error` | "两次输入的密码不一致" |
| 注册密码过短 | `st.error` | "密码至少需要 6 位" |
| 用户名已存在 | `st.error` | "该用户名已被注册" |
| 注册成功 | `st.success` | "注册成功！请登录" |
| 登录成功 | 直接跳转 | 无提示 |

**设计原则**：登录失败不区分"用户不存在"和"密码错误"，统一提示"用户名或密码错误"，防止用户名枚举攻击。

## 5. 异常处理设计

### 5.1 异常分类

| 异常类型 | 场景 | 处理方式 |
|---|---|---|
| 数据库连接失败 | MySQL 不可达 | `st.error("系统暂时不可用，请稍后重试")` + 控制台输出 |
| 用户名重复 | 注册时唯一键冲突 | `st.error("该用户名已被注册")` |
| bcrypt 内部错误 | 加盐/哈希异常（极罕见） | `st.error("系统错误，请联系管理员")` + 控制台输出 |
| 空输入 | 表单提交时未填写 | 前端校验，`st.error` 提示，不发后端请求 |

### 5.2 数据库连接管理

每次操作新建连接，`try/catch` 中 `commit` 或 `rollback`，`finally` 中 `close()`。不使用连接池（Streamlit 单用户模式，并发量为 1）。

### 5.3 日志策略

第一版不做日志文件持久化。数据库异常情况 `print` 输出到 stderr（Streamlit 控制台可见），正常操作不记日志。

## 6. 测试策略

### 6.1 测试范围

| 测试层级 | 测试内容 | 方式 |
|---|---|---|
| 单元测试 | `hash_password` / `verify_password` | `pytest` 自动化 |
| 单元测试 | `find_user` / `insert_user` / `update_last_login` | `pytest` + 测试数据库 |
| 集成测试 | 注册完整流程 | `pytest` + 测试数据库 |
| 集成测试 | 登录完整流程 | `pytest` + 测试数据库 |
| 手工验证 | UI 展示、页面跳转、登出 | `streamlit run` 走流程 |

### 6.2 测试用例清单

#### hash_password & verify_password

| 编号 | 输入 | 预期结果 |
|---|---|---|
| UT-01 | `hash_password("abc123")` | 返回 60 字符的 bcrypt 哈希字符串 |
| UT-02 | `verify_password("abc123", hash)` | `True` |
| UT-03 | `verify_password("wrong", hash)` | `False` |
| UT-04 | 两次 `hash_password("same")` | 生成不同哈希（bcrypt 随机盐），验证均返回 `True` |

#### 数据库操作

| 编号 | 操作 | 预期结果 |
|---|---|---|
| UT-05 | `insert_user("test", hash)` → 再 `find_user("test")` | 返回包含 `test` 和相同 hash 的记录 |
| UT-06 | `insert_user("test", hash)` 再次调用 | 抛出异常（唯一键冲突） |
| UT-07 | `find_user("nonexist")` | 返回 `None` |
| UT-08 | `update_last_login("test")` → 再查询 | `last_login` 字段被更新为当前时间 |

#### 注册流程

| 编号 | 场景 | 预期结果 |
|---|---|---|
| IT-01 | 正常注册（新用户名 + 合法密码） | 返回 `(True, "注册成功")`，数据库有记录 |
| IT-02 | 用户名已存在 | 返回 `(False, "该用户名已被注册")` |
| IT-03 | 密码不足 6 位 | 返回 `(False, "密码至少需要 6 位")` |
| IT-04 | 用户名为空 | 返回 `(False, "用户名不能为空")` |
| IT-05 | 密码为空 | 返回 `(False, "密码不能为空")` |

#### 登录流程

| 编号 | 场景 | 预期结果 |
|---|---|---|
| IT-06 | 正确用户名 + 正确密码 | 返回 `(True, "登录成功")`，`st.session_state["user"]` 已设置 |
| IT-07 | 正确用户名 + 错误密码 | 返回 `(False, "用户名或密码错误")` |
| IT-08 | 不存在的用户名 | 返回 `(False, "用户名或密码错误")` |
| IT-09 | 空用户名或空密码 | 返回 `(False, "用户名和密码不能为空")` |

### 6.3 测试环境

- 使用独立测试数据库 `app_auth_test`，表结构与正式一致
- 每个测试用例前后做数据清理，保证用例独立
- 测试不依赖 Streamlit session，直接调 `auth.py` 函数
- Session 相关逻辑在手工验证阶段覆盖
